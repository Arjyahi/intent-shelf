# IntentShelf Event Schema

This document explains the Phase 2 backend schema scaffold for IntentShelf.

Phase 2 is intentionally limited to:

- canonical backend entity definitions
- a clear split between offline H&M data and runtime app events
- simple example payloads to make the schema concrete

It does **not** include:

- retrieval logic
- ranking logic
- database integration
- API endpoints for these entities

## Why Two Kinds of Interaction Data Exist

IntentShelf needs two different kinds of behavioral data:

1. Historical offline data from H&M.
2. Runtime events generated by the IntentShelf app itself.

These are related, but they are not the same thing.

Historical H&M purchase data tells us what happened in the dataset before the
IntentShelf application existed. Runtime app events tell us what the
IntentShelf experience showed to a user and what that user did inside the app.

Keeping them separate avoids mixing:

- past purchases
- app impressions
- likes and saves
- session clicks
- search activity

That separation will matter later for retrieval, reranking, explanation, and
online-style logging.

## Entity Groups

### Canonical Entities

These are the app-facing entities that other systems will build on.

#### `Product`

Represents the canonical product object used by the app.

Why it exists:

- one stable product shape for retrieval, search, ranking, and explanation work
- keeps the cleaned product metadata close to the Phase 1 processed table

Planned later use:

- product detail page
- similar item retrieval
- text-based retrieval and search
- image-based retrieval once embeddings exist

#### `User`

Represents the canonical user object used by the app.

Why it exists:

- one stable user shape instead of passing raw H&M customer rows around
- leaves room for future runtime-only app users

Planned later use:

- personalization
- cold-start logic
- user-level evaluation slices

### Historical Offline Data

#### `HistoricalInteraction`

Represents a purchase from the H&M training data.

Why it exists:

- it is the clean offline signal used for recommender experiments
- it should stay distinct from app-generated events like likes or impressions

Planned later use:

- collaborative retrieval training inputs
- offline evaluation labels
- purchase-history features

### Runtime App Entities and Events

#### `Session`

Represents one app browsing session.

Why it exists:

- later session-aware reranking needs a stable session container
- app behavior often depends on current context, not just long-term history

Planned later use:

- session reranking
- session-level analytics
- grouping search, click, and impression events

#### `SessionEvent`

Represents behavior inside a session such as:

- `product_view`
- `click`
- `detail_open`
- `similar_item_click`

Why it exists:

- captures navigation and intent signals that are weaker than purchases but
  still useful
- preserves which surface and rank position produced the interaction

Planned later use:

- session-based reranking
- candidate source analysis
- product surface diagnostics

#### `SearchEvent`

Represents a query issued during a session.

Why it exists:

- search queries are a strong intent signal
- search will later be blended with recommendations

Planned later use:

- search + recommendation blending
- query-aware reranking
- search diagnostics

#### `ImpressionEvent`

Represents an item shown to a user.

Why it exists:

- ranking analysis needs to know what the system actually displayed
- clicks and saves are hard to interpret without the shown set
- later A/B-ready logging needs impression-level traces

Planned later use:

- ranking strategy comparisons
- click-through analysis
- explanation tracing by surface and source

#### `LikeEvent`

Represents a user explicitly liking an item.

Why it exists:

- this is a strong positive preference signal generated by the app
- it is different from a historical purchase and different from a passive click

Planned later use:

- preference-aware retrieval
- lightweight personalization
- explanation features

#### `SaveEvent`

Represents a user saving an item for later.

Why it exists:

- save behavior is often high intent but different from an immediate purchase
- it can later support saved-items surfaces and stronger personalization

Planned later use:

- saved-items page
- reranking features
- conversion funnel analysis

## Why Impressions Matter

Impressions answer a basic but important question:

"What did the system show before the user reacted?"

Without impression logging, later analysis can easily over-focus on clicks or
saves while ignoring the denominator. Impression events make it possible to
study:

- what was shown
- where it was shown
- at what rank
- under which ranking strategy
- from which candidate source

## Why Sessions Matter

Sessions give short-term context that historical purchases cannot capture.

A user who just searched for "black dress" and clicked several evening items is
in a different state than the same user viewed over a long time horizon.

Session grouping will later help with:

- session reranking
- search-aware browsing
- understanding surface-to-surface navigation

Small example payloads live in:
